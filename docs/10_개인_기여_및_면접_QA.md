## 예상 면접 질문과 답변 초안

### Q1. 본인 기여는 정확히 어디까지인가?

프로젝트 전체를 다 했다고 말하기보다, 제가 실제로 많이 건드린 축을 기준으로 설명하는 편이 안전합니다. 제 기준에서는 Room Builder에서 Editor로 이어지는 흐름, 프론트 구조 정리, 협업/업로드 보안 정리, 제출용 문서 체계를 맡았다고 설명하는 것이 가장 정확합니다.

### Q2. 가장 어려웠던 문제는 무엇이었나?

실시간 협업을 “보여주는 수준”이 아니라 인증과 권한까지 연결하는 부분이 가장 까다로웠습니다. REST API만 JWT를 검증해서는 충분하지 않았고, Socket.IO 연결 시점과 `join_project` 단계에서 다시 프로젝트 접근을 확인해야 했습니다.

### Q3. 왜 이런 구조를 선택했나?

FastAPI는 CRUD와 문서화에 강하고, Socket.IO는 협업용 room/event 모델을 만들기 쉬웠습니다. 프론트는 Next.js App Router와 Zustand 조합으로 페이지/상태/3D 씬을 분리하기 쉬워서 선택했습니다.

### Q4. 보안 면에서 무엇을 바꿨나?

- Socket.IO에 JWT 인증 추가
- 프로젝트 room join 권한 검사 추가
- 내부 파일 경로 노출 제거
- 보호된 다운로드 엔드포인트 도입
- 카탈로그 쓰기 API를 관리자 allowlist로 제한

### Q5. 실패했던 시도는 무엇이었나?

구조 설계, 가구 배치, 파일 처리, 협업을 한 컴포넌트/한 흐름에 많이 넣으려 하면 빠르게 복잡해졌습니다. 이후 Room Builder와 Editor의 역할을 분리하고, 프론트와 백엔드 모두 책임을 나누는 방향으로 정리했습니다.

### Q6. 다시 한다면 무엇을 바꿀 것인가?

- 관리자 권한을 allowlist가 아니라 역할 기반 권한으로 설계
- Scene / Room Builder의 렌더링 로직 추가 분리
- 중앙 로그 수집과 배포 자동 롤백까지 포함한 운영 구조 확장

## 면접에서 강조할 포인트

- 단순 기능 나열보다 “문제 -> 선택 -> trade-off -> 결과” 순서로 말하기
- 보안/협업/업로드를 함께 설명하면 기술적 깊이가 잘 드러남
- README와 `docs/09`, `docs/05`, `docs/06`을 한 세트로 준비하면 방어가 쉬움
