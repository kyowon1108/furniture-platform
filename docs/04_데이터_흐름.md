# 데이터 흐름 상세 가이드

## 전체 흐름

```text
User
  ├─ 로그인 / 프로젝트 생성 / 저장
  │      ↓
  │   REST API (JWT Bearer)
  │      ↓
  │   FastAPI + Services + DB
  │
  └─ 협업 입장 / 이동 / 잠금 / presence
         ↓
      Socket.IO (auth.token)
         ↓
      Authorized project room
```

## 1. 로그인 흐름

1. 사용자가 로그인 폼 제출
2. `POST /api/v1/auth/login`
3. 백엔드가 JWT 발급
4. 프론트는 토큰을 `sessionStorage`에 저장
5. `GET /api/v1/auth/me`로 사용자 정보 조회
6. 이후 REST API와 Socket.IO가 같은 토큰을 사용

## 2. 프로젝트 생성 흐름

1. 사용자가 프로젝트 생성 모달 제출
2. `POST /api/v1/projects`
3. 백엔드가 `Project`와 초기 `Layout(version=1)` 생성
4. 프론트는 `room-builder/{projectId}` 또는 `editor/{projectId}`로 이동

## 3. Room Builder 업로드 흐름

1. 사용자가 방 구조를 설계하고 GLB export
2. `POST /api/v1/room-builder/upload-glb/{project_id}`
3. 서버는 스트리밍 업로드 + 크기 제한 + GLB 헤더 검증 수행
4. DB에는 내부 파일 경로 저장
5. 응답은 내부 경로 대신 `download_url` 제공
6. 에디터는 이후 보호된 다운로드 API로 모델을 로드

## 4. 에디터 로드 흐름

1. 프론트가 `GET /api/v1/projects/{project_id}` 호출
2. 프로젝트 메타데이터와 `current_layout` 수신
3. `has_3d_file`, `file_type`, `download_url`, `build_mode`, `room_structure` 확인
4. `GET /api/v1/projects/{project_id}/layouts/current`로 현재 레이아웃 동기화
5. `Scene`가 `projectId` 기준 보호된 다운로드 엔드포인트에서 GLB/PLY를 fetch

## 5. 저장 흐름

1. 가구 배치/수정
2. `editorStore`가 `hasUnsavedChanges`를 true로 변경
3. 자동 저장 또는 수동 저장 시 `POST /api/v1/projects/{project_id}/layouts`
4. 새 layout version 생성

## 6. 실시간 협업 흐름

1. 클라이언트가 Socket.IO 연결 시 `auth.token` 전달
2. 서버가 토큰을 검증하고 소켓을 사용자에 바인딩
3. `join_project` 이벤트에서 프로젝트 접근 권한 검사
4. 이후 `furniture_move`, `furniture_add`, `furniture_delete`, `request_lock`, `update_presence`는
   - 인증된 사용자
   - room membership
   를 모두 만족해야 처리

## 7. 카탈로그 관리 흐름

### 일반 사용자

- `GET /catalog`
- `GET /catalog/{item_id}`
- `GET /catalog/glb/{item_id}`

### 관리자

- `POST /catalog/sync`
- `GET /catalog/list-glb`
- `PUT /catalog/{item_id}`
- `DELETE /catalog/{item_id}`
- `POST /catalog/glb/{item_id}`

관리자 여부는 `ADMIN_EMAILS` allowlist로 결정됩니다.
