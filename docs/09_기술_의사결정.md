# 기술 의사결정

## 목적

이 문서는 제출/면접에서 자주 질문받는 선택 이유를 짧게 정리한 문서입니다.

## 1. 왜 FastAPI + Socket.IO였는가

- REST API와 실시간 협업 이벤트를 같은 Python 백엔드에서 관리하기 쉬웠습니다.
- FastAPI는 CRUD, 스키마, Swagger 문서화에 유리했습니다.
- Socket.IO는 reconnect, room 개념, 이벤트 모델이 협업 편집 시나리오와 잘 맞았습니다.

## 2. 왜 `download_url`로 바꿨는가

- 초기 구조에서는 내부 파일 경로가 응답에 노출될 수 있었습니다.
- 제출/운영 관점에서는 파일 시스템 경로를 API 바깥으로 노출하는 것이 불필요하고 위험했습니다.
- 그래서 DB에는 내부 경로를 유지하되, 외부 응답은 보호된 다운로드 엔드포인트만 가리키도록 바꿨습니다.

## 3. 왜 `sessionStorage`를 기본 토큰 저장소로 바꿨는가

- 기존 `localStorage`는 세션이 끝난 뒤에도 토큰이 남아 브라우저 재사용 시 관리가 느슨했습니다.
- 제출용/면접용 설명 기준에서는 “브라우저 세션 단위 저장”이 더 보수적인 선택이었습니다.
- 그래서 `sessionStorage`를 기본으로 두고, 기존 `localStorage` 토큰은 자동 마이그레이션만 하도록 정리했습니다.

## 4. 왜 `ADMIN_EMAILS` allowlist를 넣었는가

- 카탈로그 수정/동기화 API는 일반 사용자 기능이 아니라 운영자 기능입니다.
- 별도 역할 시스템을 바로 도입하기 전 최소한의 운영 경계를 만드는 것이 우선이었습니다.
- 현재는 `ADMIN_EMAILS` allowlist로 관리하고, 이후 필요하면 RBAC로 확장하는 방향이 자연스럽습니다.

## 5. 왜 Room Builder와 Editor를 분리했는가

- 방 구조 설계와 가구 배치는 사용자의 작업 맥락이 다릅니다.
- Room Builder는 구조 생성/재질 준비에 집중하고, Editor는 배치/협업/저장에 집중하도록 분리했습니다.
- 이 분리가 GLB export 이후 에디터 진입 흐름을 설명하기도 쉽습니다.

## 6. 현재 남은 한계

- 관리자 권한 모델은 allowlist 수준입니다.
- 렌더링/충돌 처리 로직이 일부 컴포넌트에 아직 크게 남아 있습니다.
- 단일 인스턴스 배포 템플릿이라 운영 확장성은 별도 설계가 필요합니다.
